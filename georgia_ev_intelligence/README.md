## Project Overview

This system answers 50+ complex analytical questions about Georgia's EV supply chain, such as:

- *"Which EV supply chain roles are served by only a single company — single point of failure?"*
- *"Which county has the highest total employment among Tier 1 suppliers?"*
- *"Show all suppliers linked to Rivian Automotive broken down by tier and role."*

**Key design decisions:**
- **Deterministic retrieval over LLM routing** — entity extraction uses regex + real DB values, not an LLM
- **SQL-first, vector-second** — 90% of questions are answered by SQL; semantic search is a fallback
- **Zero LLM calls for retrieval** — only one LLM call happens (final synthesis), keeping latency low
- **Schema-agnostic** — new DB columns are automatically included in retrieval without code changes

## Model Stack

| Component | Model / Tool | Purpose |
|---|---|---|
| **LLM (Synthesis)** | `qwen2.5:7b` (Ollama) | Generates natural language answers from retrieved context |
| **LLM (Cypher gen)** | Same `qwen2.5:7b` | Phase 5: generates Neo4j Cypher with few-shot examples |
| **LLM (Judge)** | Same `qwen2.5:7b` | Evaluates answer quality (RAGAS LLM-as-judge) |
| **Embedding model** | `nomic-embed-text` (Ollama) | Creates vector embeddings for Qdrant semantic search |
| **SQL retrieval** | SQLAlchemy + Neon PostgreSQL | **No LLM** — deterministic filter queries |
| **Graph retrieval** | Neo4j + Cypher | Relationship queries (OEM → Supplier paths) |
| **Vector store** | Qdrant (local) | Semantic fallback + few-shot example store |
| **Web search** | Tavily API | Format 4 enrichment with real-time web results |

### LLM call count per question

| Phase | LLM calls | What for |
|---|---|---|
| Phase 4 | **1 call** | Final synthesis only |
| Phase 5 | **1–2 calls** | +1 for Cypher generation (if needed) |
| Phase 6 (planned) | **~0** | Fine-tuned model replaces LLM Cypher gen |

> **Important**: SQL for PostgreSQL is NEVER generated by an LLM. It is built programmatically by `sql_retriever.py` using SQLAlchemy filters derived from deterministically extracted entities.

## Phase-by-Phase Breakdown

### Phase 1 — Data Extraction & Storage

**Goal**: Extract Georgia EV company data from source documents, enrich with web data, and store in PostgreSQL + Backblaze B2.

**What it does:**
- Parses Georgia EV company records (company name, tier, role, county, employment, products, OEM relationships)
- Enriches company profiles using Tavily web search (news, products, OEM links)
- Stores structured data in **Neon PostgreSQL** (`gev_companies` table)
- Stores raw documents in **Backblaze B2** object storage

**Key files:**
```
phase1_extraction/
  extractor.py          ← parses source documents
  enricher.py           ← Tavily web enrichment
  db_writer.py          ← writes to PostgreSQL
  b2_uploader.py        ← uploads to Backblaze B2
```

**Database schema (key columns):**
```sql
gev_companies (
  company_name, tier, ev_supply_chain_role,
  location_county, employment, industry_group,
  ev_battery_relevant, facility_type,
  products_services, primary_oems
)
```

**Tavily usage**: ~1 API call per company × 205 companies = ~205 credits/month (free tier: 1,000/month)

**Run:**
```powershell
venv\Scripts\python phase1_extraction\extractor.py
venv\Scripts\python phase1_extraction\enricher.py
```

### Phase 2 — Vector Store & Embeddings

**Goal**: Embed all company text profiles into Qdrant for semantic search fallback.

**What it does:**
- Converts each company's text profile into a vector embedding using `nomic-embed-text`
- Stores vectors in local **Qdrant** instance
- Used as fallback when SQL/Neo4j returns no results
- Also serves as the retrieval backbone for Phase 5 few-shot store

**Embedding model**: `nomic-embed-text` via Ollama — 768-dimensional vectors

**Run:**
```powershell
venv\Scripts\python phase2_vectorstore\embed_companies.py
venv\Scripts\python phase2_vectorstore\validate_qdrant.py
```

### Phase 3 — Knowledge Graph

**Goal**: Build a Neo4j knowledge graph representing OEM → Supplier → Tier relationships.

**What it does:**
- Creates `(:Company)`, `(:OEM)`, `(:Tier)`, `(:County)` nodes
- Creates `[:SUPPLIES_TO]`, `[:IN_TIER]`, `[:LOCATED_IN]` relationships
- Enables graph traversal queries like *"Find all suppliers two hops from Rivian"*
- Loaded live by entity extractor for OEM and tier name validation

**Run:**
```powershell
venv\Scripts\python phase3_graph\graph_loader.py
venv\Scripts\python phase3_graph\validate_graph.py
```

### Phase 4 — RAG Agent Pipeline

**Goal**: Answer natural language questions using deterministic retrieval + LLM synthesis.

**This is the core of the system. Key design choices:**

#### 4a. Entity Extractor (`phase4_agent/entity_extractor.py`)
- **Zero LLM calls** — pure deterministic extraction
- Loads real DB values at startup (tiers, counties, OEMs, roles, company names) via `@lru_cache`
- Extracts: tier, county, OEM (supports multi-OEM), role (supports multi-role), employment range, product keywords, risk flags
- Intent detection: `is_aggregate`, `is_risk_query`, `is_oem_dependency`, `is_capacity_risk`, `is_top_n`

#### 4b. SQL Retriever (`phase4_agent/sql_retriever.py`)
- **Zero LLM calls** — pure SQLAlchemy
- Handles: tier filtering (exact vs compound), multi-OEM OR queries, employment range filters, county filters, industry group filters
- Schema-agnostic: new DB columns are automatically passed through

#### 4c. Pipeline (`phase4_agent/pipeline.py`)
Routing logic — decides which retrieval path to take:

```
Priority order:
1. SPOF query          → get_single_supplier_roles()
2. Company name match  → direct company lookup
3. OEM query           → primary_oems filter (OR for multi-OEM)
4. Aggregate query     → GROUP BY county SUM(employment)
5. Tier + role filter  → compound SQL filter
6. Top-N query         → ORDER BY employment DESC LIMIT N
7. Product keyword     → ilike products_services search
8. Semantic fallback   → Qdrant vector search
```

#### 4d. Streaming Synthesis (`phase4_agent/streaming.py`)
- **1 LLM call** per question
- Dynamic prompt: injects exact row count to prevent LLM stopping early
- Compact format for >10 rows (saves ~34s on large result sets)
- Company name faithfulness instruction (prevents hallucinations like "Racemark" → "Mark")
- Dynamic token budget: `max(300, min(900, rows × 35 + 200))`

**Run:**
```powershell
# Quick smoke test (10 tough questions)
venv\Scripts\python scripts\smoke_test_v2.py

# Validate full Phase 4 pipeline
venv\Scripts\python scripts\validate_phase4.py
```

### Phase 5 — Few-Shot RAG

**Goal**: Improve SQL/Cypher generation accuracy by injecting similar solved examples at retrieval time.

**What it does:**
- Stores 40 verified question→SQL/Cypher pairs in a separate **Qdrant collection**
- At query time, retrieves top-3 most similar examples using semantic similarity
- Injects them into the Cypher generation prompt: *"Here are similar questions and their SQL..."*
- Improves complex queries: aggregate, multi-hop graph, compound filter questions

**Key distinction:**
> Few-shot examples are **prompt injection at inference time** — NOT machine learning training.
> The 40 training examples and 50 evaluation questions are completely separate with NO overlap.

**Contamination protection:**
- Evaluation runs F1 (Only RAG) and F2 (No RAG) with few-shot **disabled**
- Only F3 and F4 use few-shot injection
- Contamination checker: flags eval questions with similarity >0.85 to any training example

**Run:**
```powershell
# Validate Phase 5 components
venv\Scripts\python scripts\validate_phase5.py

# Check few-shot contamination before eval
venv\Scripts\python scripts\run_format_eval.py --format 3 --model qwen2.5:7b --check-contamination
```

### Phase 6 — Fine-Tuning (Planned)

**Goal**: Replace the LLM Cypher generation call with a fine-tuned 1.5B model for <1s latency.

**Plan:**
1. Export 30 verified question→Cypher pairs from the 50-question eval results
2. Augment to 300+ training pairs via question rephrasing
3. Fine-tune `Qwen2.5-Coder:1.5B` using QLoRA on question→Cypher task
4. Replace the Phase 5 LLM Cypher call with the fine-tuned model
5. Expected: Cypher generation in <1s vs current ~15s

**Projected improvement:**
```
Current Phase 4+5:  ~20s per question (1 LLM call × 700 tokens)
After Phase 6:       ~3s per question (fine-tuned Cypher + template synthesis)
```

## Evaluation Framework

### 4 Evaluation Formats

| Format | Retrieval | Few-Shot | LLM Mode | Measures |
|---|---|---|---|---|
| **F1 — Only RAG** | SQL + Neo4j | ❌ OFF | Strict context only | Pure pipeline quality |
| **F2 — No RAG** | None | ❌ OFF | Pre-trained knowledge | What model knows without your data |
| **F3 — RAG + Pre-trained** | SQL + Neo4j | ✅ ON | Context + knowledge supplement | Full Phase 4+5 system |
| **F4 — RAG + Web** | SQL + Neo4j + Tavily | ✅ ON | Context + web results | Phase 4+5 + live web enrichment |

### RAGAS Metrics (LLM-as-Judge)

| Metric | What it measures |
|---|---|
| **Faithfulness** | Does answer use only retrieved context? (F2: N/A) |
| **Answer Relevancy** | Does answer address the question? |
| **Context Precision** | Were retrieved rows relevant? (F2: N/A) |
| **Context Recall** | Did retrieval capture all needed info? (F2: N/A) |
| **Answer Correctness** | Is the answer factually correct? (vs golden answers) |

## Key Optimizations Made

| Problem | Root Cause | Fix Applied |
|---|---|---|
| LLM stops after 1–2 rows | Static prompt didn't communicate row count | `build_answer_prompt()` injects exact row count |
| Company name hallucinations | LLM rephrased names (Racemark → Mark) | Prompt instruction: "copy names character-by-character" |
| Slow synthesis for large results | 700 token budget + verbose format | Compact list format for >10 rows, 35 tok/row |
| Column-skipping on schema changes | Hardcoded 10-column whitelist | Schema-agnostic `dict(row)` pass-through |
| Entity dict missing fields | Manual dict with hardcoded keys | `dataclasses.asdict(entities)` — automatic |
| Multi-OEM queries only matched first OEM | `break` on first OEM match | Collect all OEMs → OR-joined SQL filter |
| Model switching broken in eval | `Config` `@lru_cache` ignores env changes | `model_override` param flows through EVAgent → streaming |
| Qdrant deprecated API | `client.search()` removed in v1.9 | Migrated to `client.query_points()` |
