Format Breakdown
F1 — Only RAG

SQL + Neo4j retrieves rows from your DB. LLM reads ONLY those rows — no added knowledge. Few-shot injection is disabled. Measures pure pipeline quality.

few-shot: OFF
F2 — No RAG

No retrieval at all. LLM answers entirely from pre-training weights. Expected to score low on Georgia-specific facts — proves your DB adds value.

no retrieval
F3 — RAG + Pre-trained

Full Phase 4+5 system. Few-shot examples guide SQL generation. LLM may supplement with general EV domain knowledge. Current production config.

few-shot: ON
F4 — RAG + Pre-trained + Web

Phase 4+5 plus real-time Tavily web search results. DB context + live web news combined. Covers events not yet in your database.

few-shot: ON + Tavily

How the Evaluation Works
LLM-as-Judge Scoring

Each answer is scored by a local qwen2.5:7b model acting as a judge. It reads the question, retrieved context, and system answer — then assigns 0.0–1.0 scores for 5 RAGAS metrics. No external API required.


Bias Isolation

The 40 few-shot training examples are NOT used in F1 and F2 evaluations. This ensures the 50 evaluation questions score the pipeline fairly — the model cannot be "hinted" by a similar training example during scoring.


Key Comparisons

F2 vs F1: Proves RAG adds value over bare LLM knowledge.
F1 vs F3: Shows impact of few-shot SQL guidance.
F3 vs F4: Shows if real-time web data improves completeness.
